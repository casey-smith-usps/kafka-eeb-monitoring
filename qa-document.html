🎯 Hold Mail Architecture Q&A Preparation

Questions You'll Face & How to Answer Them

Purpose: This document prepares you for pushback, technical clarifications, and political navigation as you advocate for the proposed architecture that eliminates duplicate Hold Mail events across EEB topics.

🔥 Critical Questions (Expect These First)

CRITICAL Q1: Why is this a problem? CASI already approved the current design.
Your Answer:
CASI approved the data flow, but the current design creates duplicate events in two consumer-facing topics (EEB.Inbound and CCC.EEB.Inbound). This violates EEB's governance principle of having a single authoritative topic per event type.

Specific Risks:
  • Downstream consumers may accidentally subscribe to both topics and double-process events
  • No clear "topic-of-record" makes audit trails ambiguous
  • EEB flagged this as a governance concern to my manager
My proposal maintains CASI's approved flow while adding a governance layer that prevents consumer confusion.
CRITICAL Q2: How does your proposal differ from the current design?
Your Answer:
The key difference is topic naming and consumption patterns:
Aspect Current Design My Proposal
MDDC Topic EEB.Inbound (generic) MDDC.EEB.HoldMail.Manual (specific)
Consumer Access PTR2/MyPO might subscribe to both PTR2/MyPO subscribe ONLY to CCC topic
Topic-of-Record Ambiguous CCC.EEB.HoldMail.All (explicit)
Deduplication Each consumer implements their own CCC handles centrally
CRITICAL Q3: This sounds like more work. Why should we change course now?
Your Answer:
It's actually less work for downstream teams:

Without my proposal:
  • Every consumer (PTR2, MyPO, Process Hub) builds their own deduplication logic
  • Each team guesses which topic is authoritative
  • Audit teams can't trace which version of an event was processed
With my proposal:
  • CCC does deduplication once, centrally
  • Consumers subscribe to one clear topic
  • We document topic-of-record in governance docs
This is a one-time naming change that prevents ongoing technical debt.

🔧 Technical Questions

TECHNICAL Q4: How will CCC deduplicate events if the same Hold Mail appears from multiple sources?
Your Answer:
CCC should implement a composite key strategy:
# Deduplication Logic composite_key = f"{customer_id}_{address_zip}_{begin_hold_date}_{end_hold_date}" # Priority ranking when duplicates detected: 1. Online (USPS.COM) - most complete data 2. Call Center (1-800) - verified by agent 3. Manual (MDDC) - scanned, may have OCR errors # Tag final record with source output_event["source_system"] = "USPS.COM" # or "1-800" or "MDDC" output_event["dedupe_sources"] = ["USPS.COM", "MDDC"] # if matched multiple
This needs to be in the schema you received. Ask them to add:
  • SOURCE_SYSTEM (VARCHAR) - USPS.COM, 1-800, MDDC
  • DEDUPE_SOURCES (ARRAY) - List of all sources matched
  • EVENT_ID (UUID) - Unique identifier for this output event
TECHNICAL Q5: What if MDDC data arrives late? Will CCC reprocess?
Your Answer:
This is a valid concern. CCC needs a windowing strategy:

Option 1: Time-based window
  • CCC buffers events for 5 minutes before publishing
  • Gives time for late-arriving MDDC records to merge
  • Trade-off: 5-minute latency increase
Option 2: Watermark-based processing
  • CCC uses event timestamps (not processing time)
  • Allows out-of-order arrival within a tolerance window
  • Late events beyond tolerance trigger correction events
Option 3: Upsert pattern (my recommendation)
  • CCC publishes events as they arrive
  • If a late MDDC record matches existing online record, publish an UPDATE event
  • Consumers maintain state and apply updates
  • Trade-off: Consumers need upsert logic
Ask the team: "What's our SLA for Hold Mail processing? That determines our windowing strategy."
TECHNICAL Q6: How do we handle schema evolution if MDDC adds new fields?
Your Answer:
CCC should use a schema registry pattern:

Step 1: Schema versioning
  • Each topic has a schema version (e.g., v1.0, v1.1)
  • CCC publishes events with schema_version field
Step 2: Backward compatibility
  • New fields are OPTIONAL (not required)
  • Old consumers can ignore new fields
  • CCC maps MDDC fields to CCC schema
Step 3: Breaking changes
  • If MDDC changes field names or types, CCC absorbs the change
  • CCC publishes to CCC.EEB.HoldMail.All.v2
  • Consumers migrate on their own schedule
This is another benefit of my proposal—CCC acts as a schema abstraction layer.

🎯 Preemptive Objections

Objection: "This is overthinking it. Consumers can just dedupe themselves."
Your Counter:
That violates the DRY principle (Don't Repeat Yourself). If we have 5 consumers, we're writing deduplication logic 5 times. That's 5 places for bugs, 5 places to maintain, and 5 different interpretations of what "duplicate" means.

Centralized deduplication in CCC gives us:
  • One place to fix bugs
  • Consistent deduplication rules
  • Lower cognitive load for downstream teams
This is architecture 101—push complexity upstream, keep edges simple.
Objection: "We don't have time to refactor before go-live."
Your Counter:
I'm not proposing a refactor—I'm proposing a naming convention and consumption guideline. The code flow stays the same:

What changes:
  • Topic names (configuration change, not code change)
  • Documentation (new topic contract document)
  • Consumer subscriptions (1-line config per consumer)
What doesn't change:
  • MDDC integration code
  • CCC merge logic
  • Consumer processing logic
This is a 2-hour change that prevents weeks of debugging post-launch.
