Subject: Proposed Hold Mail Architecture - Topic Governance Clarification

Hi Team,

Following up on the Hold Mail ingestion architecture discussions and the duplicate data concern raised by EEB governance, I'd like to propose a clean solution that maintains CASI's approved flow while establishing clear topic-of-record guidelines.

Core Issue: Manual Hold Mail events from MDDC will appear in both EEB.Inbound and CCC.EEB.Inbound, creating ambiguity for downstream consumers and violating EEB's single-source-of-truth principle.

Proposed Architecture:

  1. MDDC publishes manual Hold Mail records → MDDC.EEB.HoldMail.Manual
    • Clear naming signals this is raw MDDC data
    • Marked as internal-only (not for direct consumer access)
  2. CCC consumes from MDDC.EEB.HoldMail.Manual
    • Merges with Online (USPS.COM) and Call Center (1-800) requests
    • Performs deduplication and enrichment
  3. CCC publishes consolidated data → CCC.EEB.HoldMail.All
    • Contains ALL Hold Mail requests from all sources
    • Designated as authoritative topic-of-record
  4. Downstream consumers (PTR2, MyPO, Process Hub) subscribe ONLY to CCC.EEB.HoldMail.All
    • No duplicate processing
    • No need for consumer-side deduplication
    • Clear audit trail

  • ✅ EEB Compliance: Aligns with governance principles for topic management
  • ✅ Clean Lineage: MDDC → EEB → CCC → EEB → Consumers (traceable flow)
