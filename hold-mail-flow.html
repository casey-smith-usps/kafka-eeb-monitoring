PROPOSED ARCHITECTURE: Hold Mail Data Flow - Single Source of Truth

Step Component Description Details
STEP 1 SOURCE SYSTEMS Three independent sources generate Hold Mail requests 1. USPS.COM - Online customer requests
2. 1-800 Call Center - Phone requests
3. MDDC - Manual office scans (PS Form 8076)
↓
STEP 2 MDDC PUBLISHES TO EEB
Topic: MDDC.EEB.HoldMail.Manual
MDDC publishes manual hold mail records to dedicated EEB topic
INTERNAL ONLY
Key Fields:
  • CUST_NBR
  • CUST_ONLINE_CHG
  • BEGIN_HOLD_DATE
  • END_HOLD_DATE
  • DELIVERY_ADDRESS
  • SOURCE_SYSTEM = "MDDC"
Access Control: Only CCC can subscribe
↓
STEP 3 CCC PROCESSING LAYER
Consumes from:
• MDDC.EEB.HoldMail.Manual
• USPS.COM internal systems
• 1-800 Call Center systems
CCC performs:
  • MERGE: Combines all three sources
  • DEDUPLICATE: Identifies duplicate holds using composite key
  • ENRICH: Adds source system tags
  • VALIDATE: Data quality checks
 Deduplication Strategy:
Composite Key = customer_id + address_zip + begin_date + end_date

Priority when duplicates found:
  1. USPS.COM (most complete)
  2. 1-800 (agent verified)
  3. MDDC (may have OCR errors)

New Fields Added:
  • EVENT_ID (UUID)
  • SOURCE_SYSTEM
  • DEDUPE_SOURCES (array)
  • PROCESSED_TIMESTAMP
↓
STEP 4 CCC PUBLISHES TO EEB
Topic: CCC.EEB.HoldMail.All
CCC publishes single consolidated stream with ALL Hold Mail requests
TOPIC-OF-RECORD
This topic contains:
  • All MDDC manual requests
  • All USPS.COM online requests
  • All 1-800 call center requests
Deduplicated & Enriched
Access: Public - All downstream consumers
Schema Version: v1.0
↓
STEP 5 DOWNSTREAM CONSUMERS All consumers subscribe ONLY to CCC.EEB.HoldMail.All Consumers:
  • PTR2 - Package tracking routing
  • MyPO - Customer portal
  • Process Hub - Workflow automation
  • Other downstream systems
⚠️ DO NOT subscribe to MDDC.EEB.HoldMail.Manual

KEY GOVERNANCE PRINCIPLE

CCC.EEB.HoldMail.All is the SINGLE AUTHORITATIVE SOURCE for all Hold Mail requests across USPS. This topic is the designated Topic-of-Record for downstream consumption. Consumers should NEVER subscribe directly to MDDC.EEB.HoldMail.Manual, as this is an internal topic used only for CCC ingestion.

Rationale: This architecture prevents duplicate event processing, establishes clear data lineage, centralizes deduplication logic, and maintains EEB governance compliance.

COMPARISON: Current vs Proposed Architecture

Aspect Current Design Proposed Architecture
MDDC Topic Name EEB.Inbound (generic) MDDC.EEB.HoldMail.Manual (specific, descriptive)
CCC Output Topic CCC.EEB.Inbound (generic) CCC.EEB.HoldMail.All (specific, authoritative)
Topic-of-Record ⚠️ Ambiguous - consumers unsure which to use ✅ CCC.EEB.HoldMail.All explicitly documented
Duplication Risk ⚠️ HIGH - MDDC events in both EEB.Inbound and CCC.EEB.Inbound ✅ NONE - Each event in only one consumer-facing topic
Deduplication Logic ⚠️ Each consumer implements their own (5+ implementations) ✅ Centralized in CCC (single implementation)
Consumer Access Pattern ⚠️ PTR2/MyPO might subscribe to both topics ✅ Consumers subscribe ONLY to CCC topic
Data Lineage ⚠️ Unclear which topic processed by which consumer ✅ Clear: MDDC → EEB → CCC → EEB → Consumer
EEB Governance Compliance ⚠️ Violates single-source-of-truth principle ✅ Compliant - one authoritative topic per event type
Implementation Effort Current state ✅ Minimal: Topic naming + documentation (2 hours)

BENEFITS OF PROPOSED ARCHITECTURE

Benefit Description
No Duplication Each Hold Mail event exists in only ONE consumer-facing topic (CCC.EEB.HoldMail.All). Eliminates risk of double processing.
Clear Topic-of-Record Downstream teams have explicit guidance on which topic is authoritative. No guesswork, no confusion.
Centralized Deduplication CCC handles all merge/dedupe logic in one place. Reduces complexity for 5+ downstream consumers.
Clean Data Lineage Simple, traceable flow: MDDC → EEB → CCC → EEB → Consumers. Audit-ready architecture.
EEB Governance Compliance Aligns with EEB's principle of one authoritative topic per event type. Reduces governance friction.
Reduced Technical Debt Prevents downstream teams from building their own deduplication logic. One place to maintain, one place to fix bugs.
Operational Safety Access control on MDDC.EEB.HoldMail.Manual prevents accidental direct consumption. Monitoring alerts catch misconfigurations.
Schema Abstraction CCC acts as buffer between source systems and consumers. If MDDC changes schema, CCC absorbs the change.
